> 基于 v2 重构（去容器化、adapter 层、stage 执行引擎），本期完成：
> 1. 前端适配新后端 API
> 2. 任务类型模板（YAML）+ 多 stage 编排引擎
> 3. 前端展示 stage runs 和 project 管理
>
> 后端 adapter/workspace/engine/stage 已就绪，本期重点是 **前端 + 编排引擎**。

## 二、任务编排引擎

### 2.1 内置任务类型模板

后端内置两个模板，以 Rust 常量或嵌入文件方式存储：

**single-stage**（默认，已支持）：
```yaml
name: single-stage
description: "单次 agent 执行"
inputs:
  - name: prompt
    description: "任务描述"
    required: true
  - name: agent_type
    description: "Agent 类型"
    required: true
    default: "claude-code"
  - name: model
    description: "模型"
    required: false

stages:
  - name: coding
    agent: "${agent_type}"
    needs_branch: true
    prompt: "${prompt}"
```

**feature-dev**（新增）：
```yaml
name: feature-dev
description: "开发新功能：编码 → 测试 → PR"
inputs:
  - name: requirement
    description: "功能需求描述"
    required: true
  - name: agent_type
    description: "Agent 类型"
    default: "claude-code"
  - name: model
    description: "模型"
    required: false

stages:
  - name: coding
    agent: "${agent_type}"
    needs_branch: true
    prompt: |
      根据以下需求进行开发：
      ${requirement}

  - name: testing
    agent: "${agent_type}"
    needs_branch: false
    context_from: coding
    prompt: |
      对当前代码改动进行全面测试。
      如果发现问题，输出详细的错误报告（包括错误信息、涉及文件、修复建议）。
      如果所有测试通过，输出 "ALL TESTS PASSED"。
    on_fail:
      goto: coding
      carry: error_report
      max_retries: 3
```

### 2.2 模板解析

```rust
// 新文件：engine/task_type.rs

struct TaskTypeDefinition {
    name: String,
    description: String,
    inputs: Vec<InputDef>,
    stages: Vec<StageDef>,
}

struct InputDef {
    name: String,
    description: String,
    required: bool,
    default: Option<String>,
}

struct StageDef {
    name: String,
    agent: String,          // 支持 "${agent_type}" 变量引用
    needs_branch: bool,
    prompt: String,         // 支持 "${xxx}" 变量引用
    context_from: Option<String>,
    on_fail: Option<OnFailAction>,
}

struct OnFailAction {
    goto: String,
    carry: String,          // "error_report" → 把失败 stage 的 error_report 传递
    max_retries: i32,
}
```

变量替换：遍历 prompt 和 agent 字符串，把 `${var}` 替换为 task.inputs 中的对应值。

### 2.3 任务编排执行器

```rust
// 新文件：engine/task.rs

struct TaskOrchestrator {
    db: Database,
    stage_executor: StageExecutor,
}

impl TaskOrchestrator {
    /// 执行整个 task：按模板编排 stages
    async fn execute(&self, task_id: &str) -> Result<()> {
        // 1. 读取 task 和 project
        // 2. 解析 task_type 模板
        // 3. 替换变量
        // 4. 按 stages 顺序执行：
        //    for stage in stages:
        //      如果有 context_from → 拼入上游 stage 的产出
        //      调用 stage_executor.execute()
        //      如果失败且有 on_fail：
        //        retry_count++
        //        如果 retry_count <= max_retries → goto 目标 stage，携带 error_report
        //        否则 → task 标记 failed
        //      如果成功 → 继续下一个 stage
        // 5. 全部 stage 通过 → task 标记 success
    }
}
```

**context_from 的实现**：
- 读取上游 stage_run 的 diff_patch + summary + error_report
- 拼接到当前 stage 的 prompt 前面
- 让测试 agent 知道 coding agent 做了什么

**on_fail 的实现**：
- stage 执行结果 exit_code != 0 或测试 agent 输出包含 FAIL 关键字
- 将当前 stage_run 的 agent_log/error_report 作为 context 传给 goto 目标 stage
- 重新创建一个 stage_run（run_number 递增）

### 2.4 API 改动

```
POST /api/tasks 改造：
  收到请求后：
    如果 task_type == "single-stage" → 当前逻辑（spawn stage executor）
    如果 task_type 为编排类型 → spawn task orchestrator
    都是 tokio::spawn 异步执行，立即返回 task_id

GET /api/task-types（新增）：
  返回可用的任务类型列表（name + description + inputs 定义）
  前端用这个来动态渲染 inputs 表单
```

## 三、后端补充

### 3.1 task_types 注册

可以不建 DB 表，直接在代码中注册内置模板：

```rust
// engine/task_type.rs
fn builtin_task_types() -> Vec<TaskTypeDefinition> {
    vec![
        parse_yaml(include_str!("../../task-types/single-stage.yaml")),
        parse_yaml(include_str!("../../task-types/feature-dev.yaml")),
    ]
}
```

对应文件：
```
task-types/
  single-stage.yaml
  feature-dev.yaml
```

之后扩展为从 DB 读自定义模板，但本期内置即可。

### 3.2 判定 stage 成功/失败的改进

目前只看 exit_code。编排模式下需要更智能的判定：

对于 testing stage，除了 exit_code 外，还应检查 agent 输出：
- 包含 "ALL TESTS PASSED" 或类似标记 → success
- 包含 "FAIL" / "ERROR" / 非零 exit code → failed
- 判定逻辑可封装为 `fn judge_stage_result(stage_name, exit_code, agent_log) -> StageRunStatus`
